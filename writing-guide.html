Writing microcopy

Microcopy is the term given to the short words or phrases used during transactions, for things like buttons, form labels, help text, alerts and questions.

General guidance

  • Think about your audience
  • Use sentence case
  • Keep the language simple. Don't use long or complicated words when easy or short ones will do – use 'buy' instead of 'purchase', 'about' instead of 'approximately', 'like' instead of 'such as', 'record' instead of 'index'
  • Avoid using jargon, for example technical or legal language
  • Use an active voice (not passive), for example, 'Complete the section below', or 'Check that your details are correct'
  • Use 'you' not 'they' when giving instructions or confirming a successful transaction, for example, 'Your payment has been accepted'
  • Avoid Americanisms and American spelling
  • Don't use abbreviations which could be confusing or misinterpreted – Co, Tel and No. should be written in full as county, telephone or phone, and number
  • Don't use 'please' on labels - but give polite, clear, short instructions
  • Don't use colons after labels
  • Don't add 'your' to labels, for example: 'Your name', 'Your address', unless you have a page with multiple people where you need to differentiate - for example: 'your name', 'partner name'
  • You can use contractions like 'you'll' instead of 'you will', or 'don't' instead of 'do not'

Asking for personal details

  • 'First name' not 'Christian name'
  • 'Middle names'
  • 'Last name' not 'Family name'
  • 'Age'
  • 'Date of birth' (not 'DOB')

Writing help text

Your first strategy when it comes to help text is to design a service that's so intuitive it doesn't need any. For this reason it helps to stick to interface design conventions where possible. Avoid innovation for its own sake - the real innovation is an easy to use government service.

Sometimes though, users need a little help. Here are some ways of providing it.

Inline help

Use this to provide examples for unfamiliar information requests or formats.

Progressive disclosure

This refers to help that appear on a page when the user interacts with a link, but remains hidden otherwise. It's useful for delivering important help to some users, without distracting or confusing everyone else. For this reason it's particularly useful for dealing with edge case user scenarios.

Pop-up help window

This is the nuclear option and should only be considered as a last resort. Popups or lightboxes should ONLY be used for delivering help relating to the concepts or terminology involved in a service or transaction. If you're using one to explain how to USE a service then you need to go back and make the interface more intuitive.

Transaction start pages

The start point for any nidirect.gov.uk transaction should be a start page on the nidirect.gov.uk core site. Users should not be able to jump to a later page in the service via some other means (eg Google).

The start page should be designed in conjunction with the nidirect Central Editorial Team to meet the following goals.

Describe the service

The page should include the name of the service, expressed as an action if possible ('Renew your passport', 'Claim for disability allowance', 'Book a driving test' etc.). If needed, it should include a very brief description of the service.

Set expectations

People will tend to base their expectations of what a transaction involves on their experience of other digital transactions, for example, online shopping and banking.

If your service has features that do not match these expectations then users should be informed of this as early on as possible, ideally on the start screen.

Users should be told about any financial costs or long waiting periods involved. If they'll be asked to provide relatively obscure information they’ll need to know so they can get it ready.

If there are specific eligibility requirements for the service let people know. If the eligibility requirements are complex you should discuss additional explanatory text with the nidirect Central Editorial Team at the outset.

It's better to ask a few questions up front (and explain why you're doing this) than to let people invest time and effort in a transaction only to discover part way through that they're not eligible to use it.

Transaction end pages

The end point of any transaction should, where feasible, be a page on the www.nidirect.gov.uk domain.

This page should be designed in conjunction with the nidirect Central Editorial Team and should let the user know:

  • that they have successfully completed the transaction
  • what further actions they need to perform, and when
  • what further actions they can expect from the service, and when
  • if applicable, how to provide feedback on their experience
  • who they should contact with any queries or complaints
  • about any related information and services
