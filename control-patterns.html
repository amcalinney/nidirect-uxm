Address lookup

  • The example below uses 'unobtrusive JavaScript' to present address / postcode lookup functionality
  • The lookup functionality is considered 'value-added' and is therefore only presented when JavaScript is available
  • The standard address fields can be completed with JavaScript off
  • Note the following code and components:
    • The body element has a 'no-js' class

      • <body class="no-js">
    • The address lookup div is hidden using css (i.e. the div is hidden when JavaScript is not available)

      • .no-js div#address-lookup { display: none; }
    • JQuery is used to remove the 'no-js' class (i.e. the div is shown when JavaScript is available)

      • // remove the no-js class
$('body').removeClass('no-js');
    • JQuery is used to show the select address input

      • <script type="text/javascript">
$(document).ready(function () {
    $('div#select-address').hide();
    $('button#find-address').click(function () {
        $('div#select-address').fadeIn(800);
  });
});
</script>

Progress trackers

  • Progress trackers are not mandatory, and must only used where the progress through the transaction is linear (ie no steps are skipped)
  • Used to guide the visitor through a number of steps to complete a specific process, typically a multi-step form or a checkout/payment process
  • Must be horizontally centered above the primary heading
  • Must not wrap to more than one line
  • Should be considered if there are 3 or more steps
  • If there are a large number steps (eg 8 or above) then you should reconsider the process/journey
  • If the page/window title and primary heading are repeated on each step of a multi-step form then the current step and total steps must be included

    • <title>Step 2 | Order a birth certificate | nidirect</title>
<h1 id="skip-dest">Order a birth certificate (Step 2 of 7)</h1>
  • The progress tracker can be hidden from view on the small and/or medium layouts if there is not enough screen real estate and the primary heading contains both the current and total step numbers (eg Step 2 of 6)
  • Must use the visual styling as per the examples below

Labelled tracker

A labelled tracker is preferred when:

  • all steps can be easily defined using one or two words
  • visitors can easily guess the purpose of each step from its label
  • the steps follow a familiar or known convention (eg a checkout or payment process)
  • there are a small number of steps (eg 5 or less)
  • there is enough screen real estate to present each step without the progress tracker looking cluttered, confusing or wraps to additional lines

Numbered tracker

A numbered step tracker is preferred when:

  • the individual steps cannot be easily defined using one or two words
  • visitors cannot easily guess the purpose of each step from its label
  • there are a large number of steps (eg 6 or more)

Date controls

Not all dates are the same. Choose a format appropriate for the kind of date you're asking about.

Memorable dates

For capturing memorable dates like dates of birth, the simplest option is to provide inputs to select the day and month, and then a text field to type in the year.

Memorable date example

Approximate dates

If you don't need an exact date and users are likely to struggle to remember it (for example the date you started your first job), make sure you allow users to enter approximate dates like 'June 1996'.

Search Control

  • The input control and button are styled differently to other controls
  • A smaller button and embedded label allows for a wider search control in a constrained space
  • Always use the 'glyphicon-search' icon; the magnifying-glass icon is one of the few icons which has universal recognition from users
  • Developers may use a wider input control when the anticipated search terms exceed 20 characters
